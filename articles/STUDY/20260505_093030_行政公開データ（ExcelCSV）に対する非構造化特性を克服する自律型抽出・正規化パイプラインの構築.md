# TITLE: 行政公開データ（Excel/CSV）に対する非構造化特性を克服する自律型抽出・正規化パイプラインの構築

# ABSTRACT 
ハードコードされた静的データによるリポジトリの肥大化（LLMのコンテキスト圧迫）を解消するため、行政機関が公開する非構造化データ（表計算ファイル）を継続的かつ自動的に構造化データ（JSON行列形式）へ変換するETLパイプライン「Open-Civic-Data-Harvester」を構築した。
本プロセスでは、日本の官公庁データ特有の「神エクセル（結合セル、不規則な空白行）」や「意味を持たないファイル名」といった課題に対し、純粋な行列（Matrix）変換によるトークン削減と、セル配置パターンをハッシュ化する「指紋（Fingerprint）検出」アプローチを実装した。これにより、メタデータに依存せずデータ構造からファイル種別を自律的に同定・カタログ化する基盤を確立した。

# DETAIL

### 1. リポジトリの肥大化解消とデータ基盤の分離
*   **初期課題:** モノレポ内に巨大な生データ（名簿、法案、財政データ等）がベタ書きされており、総トークン数が50万を超え、LLM開発における解析精度低下とコスト増大を招いていた。
*   **解決策:** 「アプリケーションのロジック」と「実社会のデータ」を完全に分離。外部ソースからデータを継続的に取得し、システムが読み込める形式へ洗浄・配信する専用のパイプライン（Harvester）を新設した。

### 2. サーバレスETLアーキテクチャの構築
*   **技術スタック:** GitHub Actionsをベースとし、Node.js (`tsx`, `xlsx`ライブラリ) によるサーバレス環境を構築。
*   **CI/CDの安定化:** `ts-node`環境におけるESMエラーを`tsx`への移行により解決。また、GitHub Actions上の`npm`キャッシュエラーの除外、および手動コミットとBotによる自動生成ファイルとの競合（コンフリクト）を`git pull --rebase`によって回避する堅牢なワークフローを実装した。
*   **運用フローの標準化:** 人間が取得したファイルを、グローバル標準の命名規則（`FY{西暦4桁}-{名称}.{拡張子}`）で特定ディレクトリへ投入（Push）するだけで、自動変換とAPI配信が完了するフローを確立した。

### 3. データ正規化とLLM向けトークン最適化（500万トークン問題の解決）
*   **課題:** 初期の変換ロジックでは、行政Excel特有の複雑なヘッダーにより無意味なキー（`__EMPTY_N`等）が大量生成され、出力データが500万トークンに達する問題が発生。
*   **行列化による圧縮:** `sheet_to_json`のオプションを`header: 1`に変更し、キーを持たない「配列の配列（純粋な行列）」として抽出。さらに、末尾の空セル（null, undefined, ""）の再帰的削除と空行のフィルタリングを実装。
*   **出力の二元化:** システム連携用の「インデントなし・完全版JSON」と、LLMのコンテキストプロンプト用に構造確認を目的とした「インデントあり・先頭抽出版（`.lite.json`）」を同時生成する機構を導入し、LLM連携時のトークン消費を劇的に削減した。

### 4. 構造的指紋（Fingerprint）検出機構の実装
*   **課題:** 行政データのファイル名（例: `000999084.xlsx`）やURLには意味論的（セマンティック）な規則がなく、外形情報からのデータ内容の特定や教師データの作成が不可能であった。
*   **解決策:** ファイル冒頭（20行分）のデータ存在領域をバイナリ（値あり=1, 空=0）にマッピングし、その分布パターンをMD5でハッシュ化する機能を実装。
*   **成果:** ファイル名に依存せず「レイアウトの癖（指紋）」によって、同一種別の統計データを自動的に同一の`habits/`ディレクトリへ分類・カタログ化することに成功した。これにより、未知のファイルであっても構造からデータを推論する自律処理の基礎が完成した。

# CONCLUSION
本セッションを通じて、意味を持たないファイル名や不規則なフォーマットが散在する行政の公開データから、機械可読かつトークン効率の高い行列データを持続的に抽出するパイプラインが完成した。
特に「指紋（Fingerprint）検出機構」の導入は、外部メタデータ（URLやファイル名）の信頼性が極めて低い環境下において、データ自身の内面的な「構造」をインデックス化するという非常に有効なアプローチであることが実証された。これにより、手動によるデータ名寄せの手間は排除され、今後は「特定されたハッシュと座標（Index）」を用いて、あらゆるダウンストリーム（解析スクリプトやLLM）へクリーンなデータを自動供給することが可能となった。

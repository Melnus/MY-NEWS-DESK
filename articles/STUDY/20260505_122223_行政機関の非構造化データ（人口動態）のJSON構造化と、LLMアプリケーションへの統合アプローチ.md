# TITLE: 行政機関の非構造化データ（人口動態）のJSON構造化と、LLMアプリケーションへの統合アプローチ

# ABSTRACT 
本セッションは、総務省が公開する「住民基本台帳に基づく人口、人口動態及び世帯数調査」のExcelデータ（2021〜2025年度分）をJSON形式へ構造化し、OSSリポジトリへ統合するプロセスを記録したものである。データの整合性検証、GitHubにおける適切なプルリクエスト（PR）の策定、巨大なJSONデータのシステム統合における課題（トークン制限）の解決策、および抽出用ツールのLLMモデル更新（Gemini 2.0 Flashへの移行）に関する技術的知見をまとめる。

# DETAIL

### 1. データの整合性検証と仕様の明確化
*   **数値のエビデンス:** ユーザーが構造化したデータ（札幌市、大阪市などの人口・出生・死亡数）は、当時の総務省公表データおよびマクロ的な日本の人口動態トレンド（少子高齢化による自然減）と一致しており、実データとして正確であることが確認された。
*   **データ件数のファクト:** 1年度あたりのデータ件数は「都道府県、郡、市区町村、政令指定都市の区」を含め約2,230件。2023年までは2,237件であったが、2024年度以降は浜松市の行政区再編（7区→3区）の影響により2,233件となることが確認された。これはデータの連続性を担保する上で重要なメタデータである。

### 2. 開発リソースの最適化とPRの策定
*   **実装見送りのエビデンス化:** 2020年以前のデータ（`.xls`形式での配布）および「人口移動報告」については、年度ごとのスキーマ（カラム定義やフォーマット）の差異が大きく、パース処理の自動化コストが著しく高いことが判明した。
*   **PRへの明記:** メンテナンス性と開発コストの観点から「過去分や別統計の統合は一旦見送り、今後の更新分から順次対応する」という技術的判断を明確にし、中立的かつ合理的なプルリクエスト文面を作成した。また、関連Issueへのリンク（`#24`）の適切な記法を確認した。

### 3. LLMアプリケーションにおける巨大JSONの取り扱い
*   **トークン制限の課題:** 構造化された行政データは数MBの容量を持つため、LLM（Gemini等）のコンテキストウィンドウに直接全投入するとトークン上限の超過や処理遅延を引き起こす。
*   **技術的解決策:** アプリケーション側で直接JSONを読ませるのではなく、以下のアーキテクチャの必要性が導き出された。
    1.  **DB化:** SQLiteやPostgreSQLへのインポートによるクエリの高速化。
    2.  **API化:** 必要な自治体・年度のデータのみを返す軽量な検索APIの構築。
    3.  **RAG（検索拡張生成）:** API経由で抽出した数百バイトのデータのみをLLMに渡し、回答を生成させる手法（トークン消費の抑制）。

### 4. 抽出ツールのアップデート
*   行政資料（PDF/Excel等）からJSONを抽出する社内ツール（Reality Harvester Chat）について、使用モデルを `gemini-1.5-flash-latest` から、より処理速度と精度に優れる最新の `gemini-2.0-flash` へ移行する改修を行った。

# CONCLUSION
行政機関が公開する非構造化・半構造化データ（Excel）を、機械可読性の高い構造化データ（JSON）へ変換する作業は、シビックテック（市民による技術活用）のインフラ構築として極めて高い価値を持つ。
本セッションを通じて、膨大なデータをオープンデータとして整備する手法と、それをLLMや外部アプリケーションで効率的に利用するためのアーキテクチャ（データベース化・API化・RAGの活用）の方向性が確立された。これにより、エビデンスに基づくデータ駆動型の議論やアプリ開発を促進する強固な基盤が構築されたと言える。
